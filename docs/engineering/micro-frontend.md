# 微前端

问题背景 -> 方案调研 -> 每种方案原理 -> 优缺点 -> 最终选型 -> 架构设计 -> 实际问题

## 项目背景
在项目中，我们的后台系统最初是一个单体前端应用，随着业务不断增加，不同模块由不同团队维护，同时技术栈也不统一，比如有 Vue、React 和 Umi 项目。
单体架构逐渐带来一些问题，比如代码耦合度高、部署风险大、团队协作效率低，因此我们开始调研微前端方案，希望实现系统拆分和独立部署。

## 目标
- 多团队协作
- 多技术栈共存
- 独立部署
- 降低发布风险

## 方案调研

- 隔离性

## iframe 微前端方案
### 优点
- 非常简单无须任何改造，接入成本低
- 完美隔离，js、css都运行在独立的环境
- 不限制使用页面上可以放置多个 iframe

### 缺点
- 无法保持路由状态，刷新后路由状态就丢失（这点也不是完全不能解决，可以讲路由作为参数拼接在链接后，刷新时去参数进行页面跳转）
- 完全的隔离导致与子应用的交互变得极其困难
- iframe 中的弹窗无法突破其本身
- 整个应用全量资源加载，加载太慢
- 通信复杂（postMessage）

## single-spa 微前端方案
> 通过劫持路由的方式来做子应用之间的切换，但接入方式需要融合自身的路由，有一定的局限性
- 基于类似 Web Components 进行渲染，从组件思想出发实现微前端
- 没有沙箱隔离（无沙箱机制，需要自行实现 JS 沙箱以及 CSS 沙箱）
- 样式容易冲突
- 需要自己实现很多功能

## qiankun 微前端方案
- 基于 single-spa 实现，通过自定义事件机制和路由监听，在路由变化时加载、卸载子应用。在渲染方面，利用 HTML Entry 技术，能加载子应用的 HTML、CSS、JavaScript 等资源，并进行沙箱隔离，避免子应用之间以及与主应用的全局变量冲突。
- 支持多种前端框架作为子应用；有完善的生命周期管理，便于对子应用进行管控；沙箱机制保障了应用间的隔离性。
- 配置相对复杂，上手成本较高；对子应用的改造有一定要求

## micro-app 微前端方案
- 基于 Web Components 实现，通过自定义元素（Custom Elements）、影子 DOM（Shadow DOM）、模板（Templates）和插槽（Slots）等核心概念，将前端应用程序拆分为自定义 HTML 元素。
- 微应用之间通过自定义事件进行通信，实现了应用间的解耦。

## wujie 微前端方案
- iframe + proxy + shadow DOM
- 零改造接入、极致兼容（含 IE）、多实例渲染、路由解耦、样式天然隔离，适合老项目微前端改造；

### 原理
> iframe 负责隔离,proxy 处理通信

## Module Federation
- 通过模块联邦将组件进行打包导出使用。
- 以共享模块的方式进行通信。
- 无 CSS 沙箱和 JS 沙箱。
- 需要webpack5

## 方案对比总结
| 方案              | 隔离性 | 性能 | 复杂度 | 成熟度 |
| ----------------- | ------ | ---- | ------ | ------ |
| iframe            | 强     | 低   | 低     | 高     |
| single-spa        | 弱     | 高   | 中     | 中     |
| qiankun           | 中     | 高   | 中     | 高     |
| micro-app         | 中     | 高   | 低     | 中     |
| wujie             | 强     | 中   | 中     | 中     |
| module federation | 弱     | 高   | 高     | 高     |

## 最终选型
最终我们选择 qiankun 作为微前端解决方案，因为它在生态成熟度、开发体验和隔离能力之间取得了比较好的平衡，同时对 Vue、React 等框架支持都比较好。

## 架构设计
```md
micro-base (主应用)
│
├── 数据系统 (Vue)
├── 运营系统 (React)
├── 广告系统 (Umi)
└── 供应商系统 (HTML)
```
### 主应用负责
- 路由管理
- 权限管理
- 子应用注册
- 通信
- 布局

### 子应用负责
- 业务功能
- 页面路由
- 独立部署

## 子应用切换加载对应资源（import-html.js）
importHtml 函数 功能：从指定 URL 加载 HTML 文件，提取其中的脚本代码并执行。 实现步骤：

1. 获取 HTML 内容：调用 fetchResource 函数从指定 URL 获取 HTML 内容，并将其插入到新创建的 div 元素中。
2. 提取脚本代码：getExternalScripts 函数遍历 HTML 中的所有 script 标签，对于内联脚本直接获取其内容，对于外部脚本则根据 src 属性请求脚本内容，最后通过 Promise.all 并行处理所有脚本的获取。
3. 执行脚本代码：execScripts 函数手动模拟 CommonJS 环境，使用 eval 函数执行提取到的脚本代码，并返回脚本导出的内容。
4. 返回结果：返回一个包含 HTML 模板、执行脚本函数和获取脚本函数的对象。
